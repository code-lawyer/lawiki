# 蕴含校验（claim↔引文）—— Tier-B 协议

锚点 lint（Tier-A）只证明"引文逐字存在于源文"。蕴含校验进一步证明"**旁边那句断言真的能从这条引文推出来**"——抓"引文真、但断言被脑补/拔高/歪曲"这一类（如认购→已实缴、拟→已、部分→全体、方向反了、指控→认定）。它是铁律 **INFERRED（不脑补）** 的机器探测器。

## 三步

**① 抽取（确定性，`extract_claims.py`）**
把 wiki 里每条带锚点的事实陈述拆成 `(page, claim, source, quote, context)`。**每个锚点配它紧前的那段子断言**（锚点引它前面的话），一行多锚点则各管各的子断言——不是整行。`context` 是引文在源文里前后约 120 字的窗口（判官靠它，不喂全文）。跳过标题与 `>` 开头的分析 callout（已显式标注的 INFERRED）。

```
python lint/extract_claims.py <案件根目录> > worklist.json
```

**② 判定（必须用「换实例」的 LLM）**
把清单交给一个**全新的 LLM 实例**（Claude Code 下用子代理；或换一个模型），逐对判定。两条硬约束：
- **换实例**：绝不让写这页的同一个 agent 判自己——它已经信了自己的断言。
- **只喂这一对 + 一小段源文上下文**：给"引文 + 该引文在源文前后各约 100–150 字的窗口"，**不喂全文、不许用外部知识**。光给引文会"上下文饿死"误报（如「壹佰万元」无"注册资本"字样被判信息不足）；给全文又让它有空间替断言圆场。窗口是折中。

判官 prompt（照用）：

> 你是严格的法律文书引文蕴含判官。仅凭给定的**引文（及其源文上下文窗口）本身**判断能否支持断言，不许用外部知识或其他对的信息替断言圆场。每条三选一：
> - **支持**：引文直接、完整支持断言，断言没有超出引文。
> - **不支持**：引文与断言矛盾，或断言把引文歪曲/拔高（认购/价款≠已实缴/已转账；拟≠已；部分≠全体；方向反了；指控≠认定）。
> - **信息不足**：引文为真，但不足以支持断言全部（断言含引文未提到的人数/合计/"全体一致同意"等）。
> 输出：表格（序号|判定|一句话理由）+ 单列所有"不支持/信息不足"的序号作为待复核清单。只判定，不改写、不建议。

**③ 分流（triage，不自动改）**
- **不支持** → 几乎必是真错（歪曲/拔高），回去把断言改成忠实于引文的说法，或降级为 `> [!note] 分析`。
- **信息不足** → 要么换一条能完整支持该断言的引文，要么拆断言（见下规则），要么标"未核验"。
- 判定是**筛子不是闸门**：会漏判会误报，只把可疑项捞给人复核，**绝不自动放行、绝不自动改写**。

## 由此得到的写作铁规则

**一条断言 ← 一条能完整支持它的引文。** 别把多个事实塞在一条锚点下（炎凰时间线最初就犯了：把"日期+轮次+金额+注册资本"挤在一条引文下，而引文只证明其中一项）。一句话里有几个独立事实，就挂几条各自够用的锚点。

## 局限

- 是**网不是保证**；必须换实例；窗口大小要调（太小饿死、太大放水）。
- 抓不了**遗漏**（断言忠实但不完整）——那归覆盖率/人工。
- 法律**定性是否正确**仍归人。
