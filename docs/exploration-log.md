# lawiki 探索日志

按时间记录设计探索与关键决策（每一步为什么这么定）。最新在下。设计正文见 `docs/superpowers/specs/`，本文件是"过程与理由"。

## 2026-06-16 起：从 makeitdown 到 lawiki

- **缘起**：makeitdown（上游，已成熟）只负责"各式文件 → LLM 可读 md"。下一步是用 LLM 把 md 建成**可控、可溯源的法律案件 wiki**。
- **定位决策**：lawiki 是**独立项目**、成果是**一个可被不同 agent 用的 skill**（不是 CLI、不是 RAG），通过 makeitdown 的稳定输出契约（frontmatter `source`/`quality: suspect`、report.json）对接。
- **形态决策（方案 A）**：放弃"skill + 每案 CLAUDE.md 两层 + 模板目录"，收敛为**单文件 `SKILL.md`** 承载全部（流水线/脚手架/ingest/铁律/锚点/四类页面格式）。理由：用户质疑"不就一个 skill 吗"，确实如此——验证脚手架≠交付物，别把活撑大。
- **领域模型（用户即法律专家定）**：四个核心板块 **案件主体 / 法律关系 / 法律事实 / 时间线**（一类一目录），主体=md 里能找到的一切身份信息，法律关系按中国大陆案由，法律事实=带证据的法律意义事实点，时间线=把法律事实按时序串起。概述/争议焦点砍掉。
- **锚点格式**：`〔来源: _md/路径：「逐字上下文片段」〕`，**不用页码**（用户：带上下文我自己能定位），逐字片段同时满足"要害逐字"。

## 2026-06-16：真实全流程实测（炎凰数据融资案，35 文件）

- makeitdown 把 35 文件全转成功（含本地 PaddleOCR 处理扫描件）；lawiki 照 skill 建出 13 页 wiki、59 锚点。
- **发现（已记 memory）**：makeitdown 质检漏检 **OCR 局部数字损坏**（容腾 `RMB2O,OOO,OO<`），因 garbled-ratio 按全文算被稀释。靠跨源交叉（决议表格）+ 算术勾稽才抓出——法律场景高危。

## 2026-06-17：Obsidian 为输出基准

- 用户确定 Obsidian 为查看器。实测：标准 markdown 链接 Obsidian 也进图谱，但 wikilink 更稳。
- 决策：产出改 **Obsidian Flavored Markdown**——交叉引用 `[[]]`、分析/冲突用 callout、frontmatter 即 properties（tags/aliases）；**来源锚点保持纯文本**（引证非导航、且要保持可机检）。

## 2026-06-17：铁律重构为三类标注（借鉴 graphify）

- 看了 graphify（YC 项目，给语料建可查询知识图谱）。判断：**借它的形式（置信标签 EXTRACTED/INFERRED/AMBIGUOUS），不借它的立场**（它主动造推断边、走云端 LLM 抽取——与法律"不接受模糊"相反）。
- 五条铁律 → **三类标注 + 一条逐字锚点硬底线**：EXTRACTED（原文直取，必挂逐字锚点，唯一能当事实直述）/ INFERRED（推断，显式标注隔离）/ AMBIGUOUS（存疑，必打标）。等价覆盖原五条，更利落。

## 2026-06-17：取证模式（bbox）评估——决定不做

- 评估"保留 OCR bbox、点结论高亮原扫描件"。结论：**价值压倒性在人/审计一侧，对 LLM 近乎为零**；bbox 唯一不冗余的价值是核验 OCR 扫描件的真像素，但需独立查看器 App，且 docx 无坐标。**不做全套**，留作仅 OCR 路径的远期。
- 关键澄清（回应用户追问）：**现方案已能定位"录入错误"**（锚点把引文摆在结论旁，矛盾当场可见）；bbox 只是 OCR 核验那一步的人体工学加速，且静默替换它也不自动抓——勾稽/交叉才抓得到。

## 2026-06-17：检测分层 + lint（确定性闸门）

- 确立检测策略：**能确定性就确定性（Tier A）→ 不能才窄域换实例 LLM（Tier B）→ 法律定性归人（Tier C）**。
- **Tier A 已实现** `lint/lint_wiki.py`：① 锚点存在（EXTRACTED 硬底线机检）② 死链 ③ 时间线顺序；④ 覆盖率（警告）。
  - **校准教训**：逐字匹配必须**归一化格式噪声**（空白/换行/全半角/千分位逗号/markdown/表格管道/HTML 标签），保留数字文字精确。炎凰上 40 假阳性逐一定位归零，期间**抓出 1 条把 OCR 打乱表头拼成连续引文的真锚点**。
  - 覆盖率正确暴露**整类未 ingest 的实质文件**（股东协议/章程/决议）。
  - 决策：lawiki 自此**有代码**了——"纯 skill"在验证这半让步，因为这正是 skill 做不了的确定性部分。

## 2026-06-17：拒绝"LLM 自动修到一致"的自愈循环

- 用户提议：LLM 自审一致性、不行就重走 makeitdown/换 OCR，三轮不行再标红。
- 评估：**核心直觉对（一致性/勾稽是最强网），但朴素版高危**——"修到一致"给模型**伪造一致**的动机（偷改加数凑总），这正是幻觉。护栏：检测用确定性代码（非 LLM 自评算术）；修复只许"重取来源"绝不"调数值"；区分提取错 vs 真实矛盾（后者只暴露不修）；法律场景**先"检测+人确认"，不轻易自动修**。

## 2026-06-17：Tier-B 蕴含校验（claim↔引文）—— 本轮

- 目的：锚点 lint 证明"引文真"，蕴含校验证明"断言能由引文推出"，合起来彻底封死 EXTRACTED；抓"引文真但断言被脑补歪"。
- 实现：确定性抽取 `extract_claims.py`（拆 `(断言,引文,来源)`）+ **换实例 LLM 判官**三分判（支持/不支持/信息不足），协议见 `lint/entailment_check.md`。
- **真实验证（炎凰 12 真实 + 3 植入歪曲）**：植入的 3 条（认购→已实缴、决议→全体一致同意、认购→已转账）**全被判不支持**；忠实条放过；**还抓出我自己 4 条"信息不足"**——把多事实塞一条锚点下、引文只撑起一部分。
- **由此两个产出**：① 写作铁规则"**一条断言 ← 一条能完整支持它的引文**"；② 设计精化"判官喂**引文 + 一小段源文上下文窗口**"以避免上下文饿死误报。
- 状态：抽取器 + 协议 + 验证完成；判官是 LLM 运行时（子代理），非可提交的确定性工具。

## 2026-06-17：抽取器 context 窗口 + 数字勾稽闭合（①+②）

- **共享归一化**：把 `norm` 抽进 `lint/_normalize.py`（含带索引映射的 `norm_with_map`），lint 与抽取器共用，避免漂移。
- **① context 窗口**：`extract_claims.py` 每对加 `context` 字段——用 `norm_with_map` 在源文里定位引文、映射回原文下标、截前后约 120 字（折叠空白）。判官从此拿"引文+上下文"，治上一步发现的上下文饿死误报。
- **② 勾稽闭合**：lint 新增第 4 类确定性检查，解析页面 `> [!check] a + b == c` 断言，**ast 安全求值**（只许 +−× 与数字，绝不 eval），行尾中文说明自动忽略。给炎凰 pre-A/A1/A1+ 三页加了真实勾稽断言。
  - **归类澄清**：数字勾稽本属"代码推不出关系"的 Tier-B，但靠 `[!check]` 这一**写作约定**显式声明等式后，校验本身是确定性的——故归入 Tier-A。
  - **调试**：右操作数前导空格让 `ast.parse(mode="eval")` 报 IndentationError，strip 修复。17 测试全过，炎凰 0 违规。

## 2026-06-17：测试效果——修 4 条"信息不足"锚点

- 拿上一次蕴含判官标的 4 条"信息不足"，**带 context 重判**：
  - context 窗口**救回**了"裸引文饿死"的 [决议日期]（→支持）；
  - 还**揭出真错**：[1] 我写"公司**设立时**"，源文是"**现有**注册资本"——改成"pre-A 轮增资前 现有注册资本"。
  - [A轮]、[股东名册] 仍信息不足——真·把多事实塞一条引文。
- 按"**一断言一引文**"拆开时间线锚点。但发现**抽取器仍把整行当每个锚点的断言**——改 `extract_claims.py`：**每个锚点配它紧前的子断言段**（锚点引它前面的话），而非整行。这比"强制一行一锚点"更对路。
- 重判 3 条修正：全部**支持**。**诚实注脚**：[增资后注册资本 1,538,461] 的支持部分靠手喂 context 时加的等价注解；raw 管线里它是**边界情形**（"1,538,461 即 A 轮增资后"是个正确的轻推断），恰踩 EXTRACTED/INFERRED 边界——不夸大为完美通过。
- 产出：抽取器 per-segment 改进（+测试）；炎凰时间线 3 条锚点修正（案件目录，未入 git）。

## 2026-06-17：蕴含判定接成 ingest 第 9 步

- 把蕴含校验从"独立工具"接成 **ingest 收尾的自动一步**（SKILL.md 第 9 步）：抽取 → 派全新子代理判官三分判 → ingest agent **有界修复**（不支持改忠实/降级分析；信息不足换引文/拆开/标未核验；**只许改忠实绝不编造**）→ 重跑 lint+重判 ≤3 轮 → 仍判不过的**显著上报用户**，不静默通过。
- 分工守住：**判官换实例、只判不改**；**修复由 ingest agent 做**；自动修受"只许改忠实"约束，未解决必上报——既自动又不洗白幻觉。增量 ingest 时第 9 步可只判新增/改动锚点省成本。
- 纯指令改动（抽取器+协议已就绪）；同步 README/spec(§10/§12)/本日志。
