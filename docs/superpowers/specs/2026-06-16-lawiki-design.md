# lawiki v1 设计

> 状态：已批准，待写实现计划。
> 日期：2026-06-16

## 一、定位与成果形态

lawiki 是一个**独立项目**，目标是**一个可被不同 agent 识别和使用的 skill**——把某个法律案件的大量资料，整合成一个**可控、可溯源**的案件 wiki。

成果物**是 skill（+ 它自带的模板与约定），不是 CLI 工具、不是 Python 包**。"可控"不依赖任何二进制程序，而依赖 skill 本身写得足够自包含、约束足够硬、足够可移植。

上下游分工：

- **makeitdown**（上游，已有的独立 CLI 工具）：把 `原始资料/` 里各式文件转成 LLM 可读的 `.md`。
- **lawiki**（本项目，skill）：用 LLM 把这些 `.md` **归档/整合**进案件 wiki。

两者通过 makeitdown 的**稳定输出契约**对接（见第五节），由 agent 照 skill 编排串成一条流水线。

### 可移植性（一等要求）

- skill **自包含**：模板、引用都打包在 skill 目录内，任何 agent 加载该 skill 即获得全套能力，不依赖外部程序。
- `SKILL.md` 的 frontmatter（`name` + `description`）写清晰的**触发语义**，让 LLM 一眼判断"何时该用它"。
- skill 正文写成 **agent 无关**的指令，不绑定某个 agent 的专有机制。
- `README.md` 说明在 Claude Code / Copilot（`SKILL.md`）与 Codex（`AGENTS.md`）等不同 agent 下如何挂载。
- 每个案件 wiki 内的 schema 文件默认命名 `CLAUDE.md`，skill 注明 Codex 等用 `AGENTS.md` 等价物。

## 二、磁盘三层结构（每个案件）

沿用 llm-wiki 的"来源不可变"原则：

```
某案件/
  原始资料/      # 用户丢入的法律文件（不可变，真相之源）
  _md/           # makeitdown 输出（不可变的"来源层"）：.md + frontmatter + report.json
  wiki/          # agent 拥有并维护的 wiki
    CLAUDE.md    # 该案件 schema（详细领域约定 + 页面模板；第二层约束）
    index.md     # 全部页面目录
    log.md       # append-only 操作日志
    案件主体/
    法律关系/
    法律事实/
    时间线/
```

`_md/` 全程是中间产物，用户不必关心——这就是"对用户无感"。

引用锚点指向 `_md/xxx.md`，再经其 `source` frontmatter 溯回 `原始资料/` 的原件——**两跳溯源链**，满足法律场景的可追溯要求。

## 三、四个核心板块

均为"**一类一目录、多页**"。不设"概述"与"争议焦点"页。

### 1. 案件主体/

每个主体一页。收录 LLM 在 `_md/` 文件堆中找到的、与该主体有关的**一切**信息：

- 身份信息（如合同中录入的当事人身份证号）；
- 代理人 / 法定代表人（如判决书中列出的当时律师）；
- 主体资格、关联关系等。

凡能找到与案件主体相关的内容都归入此处。每条信息挂来源锚点。

### 2. 法律关系/

按**中国大陆法律**确定案件归属类别，每组法律关系一页：

- 商事：股权纠纷、合同纠纷、商标案件、知产案件等；
- 民事：借贷纠纷、房产纠纷等；
- 刑事：按罪名区分。

写明类别与请求权基础 / 法律依据（法律依据并入此处，不单独成页）。

### 3. 法律事实/

LLM 根据知识库内容**分析提炼**出的、有法律意义的事实点，每个事实一页，**必须挂证据支撑**。例如：

- "欠债的事实" ← 欠条、转账记录；
- "发生投资的事实" ← 入股协议、投资款转账记录。

证据不单独成页——它以来源锚点的形式嵌在法律事实（及案件主体）页中。

### 4. 时间线/

把上述法律事实**按时间顺序**整理、串成时间线。每条时间线条目指回对应的法律事实页与来源锚点。

## 四、五条铁律（写进 skill，常驻 agent 注意力）

这是"可控"的命门。短、绝对、可检查：

1. **零来源不可写**：wiki 中每一句事实陈述都必须挂一个来源锚点（指到具体的源 md、能定位到页/位置）。没有来源 → 不许进 wiki。
2. **不推断、不脑补**：只写文档明确说了的；任何分析/推论必须显式标注为"分析"，与事实物理隔离。
3. **可疑来源必须标记**：frontmatter 带 `quality: suspect` 的 md，引用时一律标"未经核验"，绝不当作既定事实。
4. **矛盾只暴露、不私自调和**：新来源与已有内容冲突时，标记冲突，绝不静默覆盖。
5. **法律要害逐字照录**：日期、金额、当事人名、条款原文一律逐字引用，不许转述。

## 五、makeitdown 对接契约

lawiki 依赖 makeitdown 的稳定输出：

- 镜像输入结构的 `.md` 树；
- 每个 `.md` 带 frontmatter：`source` / `source_type` / `engine` / `pages` / `converted_at`；可疑文件额外带 `quality: suspect` + `warnings`；
- 一份 `report.json`（succeeded / warned / failed / skipped 台账）。

对法律场景的价值：`source` 是**溯源锚点**；`quality: suspect` / `warnings` 是**质量信号**，ingest 时据此决定某来源能否直接采信（对应铁律 3）。

## 六、引用锚点格式（机器可校验）

wiki 中每条事实后挂固定格式锚点：

```
合同签订于 2021 年 3 月 5 日 〔来源: _md/合同/采购框架.md#p3〕
```

固定 `〔来源: <md路径>#p<页>〕`，**正则可扫**。案件主体属性、法律事实、时间线每条都必须有。该格式在 v1 即钉死，使铁律 1 从"软约束"成为"可机检"——留给 v2 的 lint 自动扫描"哪句话没挂来源"。

## 七、编排流（A 方案，对用户无感）

用户只做一件事：把文件丢进 `原始资料/`，告诉 agent "处理一下"。skill 驱动 agent：

```
原始资料/ ──[skill 调 makeitdown CLI]──▶ _md/ ──[skill 读 frontmatter 按质量信号 ingest]──▶ wiki/
```

makeitdown 这一步是 agent 去跑 makeitdown 自己的 CLI（上游独立工具），与"lawiki 本身不是 CLI"不冲突。

## 八、ingest 工作流（写进 skill 的操作骨架）

每个新来源：

1. 读该 md 及其 frontmatter；
2. 若 `quality: suspect` → 标"未核验"（铁律 3）；
3. 抽取主体信息，归入 `案件主体/`；
4. 提炼有法律意义的事实点及其证据，归入 `法律事实/`；
5. 判定 / 更新案件的 `法律关系/`；
6. 把事实按时序并入 `时间线/`；
7. 维护交叉引用、更新 `index.md`、追加 `log.md`。

新建案件时，agent 照 skill 把 `templates/case-wiki/` 拷贝成 `wiki/`，并建 `原始资料/`、`_md/`。"可复现"来自固定模板 + skill 写死的结构，而非某个程序。

## 九、项目仓库结构（纯 skill 包，无代码）

```
lawiki/
  skill/lawiki/
    SKILL.md                  # 短 skill：触发语义 + 编排骨架 + 五条铁律
    templates/case-wiki/      # agent 直接拷贝的脚手架：
                              #   CLAUDE.md（案件 schema：详细约定 + 页面模板，含锚点格式占位）
                              #   index.md、log.md（空骨架）
                              #   案件主体/、法律关系/、法律事实/、时间线/ 四目录
                              #   每类一个页面模板
    references/               # 选填：详细约定，skill 按需链接、不常驻注意力
  README.md                   # 给人看：这是什么、如何在不同 agent 下挂载
  docs/                       # 设计文档（本 spec 等）
```

无 `src/`、无 CLI 命令、无 pytest。

## 十、范围

**v1（本 spec）**：makeitdown 编排 + 三层结构/schema + 四个法律板块 + ingest 工作流 + 五条铁律 skill + 案件脚手架模板。

**v2（暂不做）**：query（对 wiki 提问并带引用回答）、lint（自动健康检查：矛盾/孤立页/陈旧声明/扫描无锚点的事实）、wiki 搜索工具。

## 十一、验证方式

v1 几乎无确定性代码，故不设单元测试。验证 = **端到端人工验收**：

- 用一个样例小案件（两三个假法律文件，覆盖"含可疑 OCR 来源"与"两来源相互矛盾"两种情况）；
- 让 agent 照 skill 端到端走一遍：转换 → 建出 `案件主体/法律关系/法律事实/时间线` + `index/log`；
- 对照人工验收清单核查：
  - 每条事实是否都有 `〔来源: …〕` 锚点（铁律 1）；
  - 可疑来源是否标了"未核验"（铁律 3）；
  - 矛盾是否被暴露而非静默调和（铁律 4）；
  - 要害信息（日期/金额/姓名）是否逐字（铁律 5）；
  - 目录结构与模板是否一致。

引用锚点的固定格式保持机器可校验，留给 v2 的 lint 自动扫描。
